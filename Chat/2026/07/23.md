
현재 프런트엔드는 `/api/equipment`처럼 상대 경로로 API를 호출하므로, 도메인과 HTTPS로 바뀌어도 프런트엔드 수정량은 적습니다.

## 1. 먼저 결정할 사항

가장 먼저 “어디서 접속할 것인가”를 정해야 합니다.

| 사용 목적 | 권장 방식 | 필요한 것 |
|---|---|---|
| 외부에서도 접속 | 공개 도메인 + 공인 인증서 | 도메인, DNS, 공유기 포트포워딩 또는 터널 |
| 집/사무실 내부에서만 접속 | 내부 도메인 + 내부 CA 인증서 | 내부 DNS 또는 각 기기의 hosts 설정, CA 신뢰 등록 |
| VPN 사용자만 접속 | VPN + 내부 HTTPS | VPN 구성, 내부 도메인 또는 내부 CA |

외부에서도 사용할 계획이라면 `equipment.example.com` 같은 도메인을 준비하고 DNS의 A 레코드를 집/사무실의 공인 IP로 연결하는 구성이 가장 관리하기 좋습니다. Caddy는 도메인, 80/443 포트 접근이 준비되면 인증서 발급·갱신·HTTP→HTTPS 리다이렉트를 자동으로 처리할 수 있습니다. [Caddy HTTPS 안내](https://caddyserver.com/docs/quick-starts/https)

현재 주소인 `192.168.0.166`은 사설 IP입니다. 내부 전용이라면 공인 인증서보다는 Caddy 내부 인증서를 쓰는 편이 현실적입니다. 대신 접속하는 PC·모바일마다 Caddy의 루트 인증서를 신뢰하도록 설치해야 브라우저 경고가 사라집니다. [Caddy 내부 HTTPS 안내](https://caddyserver.com/docs/running)

참고로 Let’s Encrypt는 2026년부터 IP 인증서도 지원하지만, 짧은 6일짜리 인증서이며 자동 갱신·검증 구성이 까다롭습니다. 사설 IP `192.168.x.x`를 위한 일반적인 해법으로는 추천하지 않습니다. [Let’s Encrypt IP 인증서 안내](https://letsencrypt.org/2026/01/15/6day-and-ip-general-availability.html)

## 2. Flask 앱을 운영 방식으로 변경

현재 `app.run(..., debug=True)`는 개발용입니다. 운영 전환 시에는 다음을 진행합니다.

- `debug=True`를 제거하거나 개발 환경에서만 켜도록 분리
- Flask 내장 서버 대신 Gunicorn으로 실행
- 앱은 외부가 아닌 `127.0.0.1:8000`에서만 수신
- SQLite 파일이 실행 위치에 따라 달라지지 않도록, 서비스의 작업 경로를 고정
- `requirements.txt`를 만들어 Flask와 Gunicorn 버전을 기록
- 시스템 부팅 후에도 실행되도록 `systemd` 서비스로 등록

예시 실행 명령은 다음과 같습니다.

```bash
cd /home/<사용자>/Mini-Server-Web-EqMgmt
python3 -m venv .venv
. .venv/bin/activate
pip install flask gunicorn

gunicorn --workers 2 --bind 127.0.0.1:8000 app:app
```

이 단계가 끝나면 브라우저에서 포트 `5000`으로 접근하지 않습니다. Gunicorn의 `8000` 포트도 서버 내부에서만 접근 가능해야 합니다.

## 3. Caddy로 HTTPS 종료

Linux Lite 서버에 Caddy를 설치하고 `/etc/caddy/Caddyfile`에 아래처럼 설정하는 방식을 권장합니다.

공개 도메인을 사용할 경우:

```caddyfile
equipment.example.com {
    reverse_proxy 127.0.0.1:8000
}
```

이 설정만으로 Caddy는 HTTPS 인증서 발급과 갱신, `http://equipment.example.com`의 HTTPS 리다이렉트를 관리합니다. Caddy는 프록시 과정에서 `X-Forwarded-*` 헤더도 전달합니다. [Caddy reverse_proxy 문서](https://caddyserver.com/docs/caddyfile/directives/reverse_proxy)

내부망 전용이라면, 예를 들어 내부 DNS에서 `equipment.home.arpa`를 `192.168.0.166`으로 해석하게 한 다음 아래처럼 구성합니다.

```caddyfile
equipment.home.arpa {
    tls internal
    reverse_proxy 127.0.0.1:8000
}
```

그 다음 각 클라이언트에 Caddy 내부 CA의 루트 인증서를 신뢰하도록 설치해야 합니다. 단순히 자체 서명 인증서를 만들고 브라우저 경고를 무시하는 방식은 권장하지 않습니다.

## 4. 네트워크와 방화벽

외부 공개 방식일 때 필요한 작업입니다.

- 공유기에서 TCP `80`, `443`을 미니서버 `192.168.0.166`으로 포트포워딩
- 서버 방화벽에서 `80/tcp`, `443/tcp`만 허용
- 기존 `5000/tcp` 외부 공개는 중단
- 도메인의 DNS A 레코드가 공인 IP를 가리키는지 확인
- 통신사 환경이 CGNAT라 포트포워딩이 불가능한지 확인

외부 공개가 필요 없으면 포트포워딩을 하지 않고, 방화벽도 내부망 대역에서만 443 접근을 허용하는 편이 더 안전합니다.

## 5. HTTPS와 별도로 반드시 점검할 보안

HTTPS는 전송 구간을 암호화할 뿐, 누가 장비 데이터를 수정할 수 있는지는 통제하지 않습니다. 현재 API는 인증 없이 조회·등록·수정·삭제가 가능합니다. 외부 공개 시에는 특히 다음이 필요합니다.

- 로그인 및 사용자 권한 관리
- 인증되지 않은 사용자의 `POST`, `PUT`, `DELETE` 차단
- 세션 기반 로그인이라면 CSRF 방어
- 입력값 검증과 오류 처리
- 서버 방화벽에서 필요한 포트만 허용
- SQLite DB 파일의 권한 제한 및 정기 백업
- 접근 로그와 Caddy/Gunicorn 서비스 로그 확인
- 보안 헤더 적용

보안 헤더 중 HSTS는 HTTPS 전환이 완전히 안정된 뒤 적용해야 합니다. 잘못 적용하면 브라우저가 HTTP 접속을 강제로 차단해, 설정 오류 시 복구가 번거로울 수 있습니다. 또한 현재 Tailwind CDN과 인라인 JavaScript를 쓰므로, 강한 CSP는 테스트 없이 바로 적용하면 화면이 동작하지 않을 수 있습니다.

- 실제 적용 결과와 제안 단계의 내용은 별도 표기로 구분
- 민감 정보만 예외적으로 자리표시자로 치환하고, 그 사실을 기록
